Cold Start Pipeline

Deterministic 14-step build pipeline for constructing Figma component sets from scratch. Every phase is one plugin invocation, every gate is a halt-on-mismatch assertion.

Pipeline v3.2.0+cold-start-bound 14 Steps (S0 – S13) 13 Phases (0 – 12) 24 Rule Checks

Pipeline Overview

The cold-start pipeline builds a complete Figma component set deterministically from two input files: a component spec (what to build) and a rules file (how to build it). Each phase maps to exactly one Figma plugin invocation. Between phases, cross-call assertions verify the previous phase's output before allowing the next phase to proceed. If any assertion fails, the pipeline halts — no silent failures, no post-build patches.

S0 Input Hashes ↓ S1 Library Imports ↓ S2 Placeholders ↓ S3–S4 Subcomponents ↓ S5 Parent Variants ↓ S6–S8 Instances + Properties + Bindings ↓ S9–S11 Verification + Text Styles + Exposure ↓ S13 Full Audit (24 checks)

Core Invariants

These rules apply across every phase. Violating any one is a halt condition.

R26 + R28: Every auto-layout container MUST be created via mkAutoLayoutFrame(). Never call resize() on auto-layout containers — it silently overrides primaryAxisSizingMode from AUTO to FIXED.
R25: Every INSTANCE node whose spec declares exposeAtParent: true must have isExposedInstance = true, recursively through the spec tree.
R12: Plugin API blocks componentPropertyReferences on instance sublayers. Button labels are only reachable via isExposedInstance — there is no workaround.
0 Page Initialization S0

Creates (or finds) the dedicated Figma page where all component artifacts will live. This is the clean slate — nothing else runs until the page exists and is set as the active page. The pipeline version number and input file checksums are recorded here as the build's fingerprint.

Why a dedicated page? Isolating the build on its own page prevents artifacts from landing on the wrong page (e.g., the Cover page) and ensures the AABB overlap check in later phases works against a known artifact set.

Operations

  1. Search figma.root.children for a page named 'Monocore Card N' (where N is the build iteration number).
  2. If not found, create a new page via figma.createPage() and set its name.
  3. Call figma.setCurrentPageAsync(page) to make it the active page.
  4. Compute SHA-256 hashes (16 hex char prefix) of both input files and embed in the build ledger.

2 Local Placeholders S2

Creates five small placeholder components that serve as default content for slot properties. When a designer drags a fresh card onto the canvas, these placeholders fill the empty slots with recognizable shapes — a 48-pixel thumbnail square, two 16-pixel icon squares, and two slot region indicators — so the card never appears broken out of the box.

Operations

  4. Store a placeholders map keyed by slot name for Phase 5 lookup.

3 Card Media Subcomponent S3

Builds the Card Media component set — a 3-variant component (Image, Icon, None) that controls what appears in the card's thumbnail region. This is a separate component set from the main card, connected via instance swap. Each variant holds a single slot frame where image or icon content can be inserted.

Cross-call slot promotion: After the SLOT property is bound, the slot frame's type changes from FRAME to SLOT — but only on the next plugin invocation. The pipeline accounts for this one-call delay.

Operations

  1. Create 3 COMPONENT nodes, each with a single child FRAME named MediaFrame, named for the Pattern axis: Pattern=Image, Pattern=Icon, Pattern=None.
  2. Call figma.combineAsVariants(mediaVariants, page) and name the set 'Card Media'.
  3. Configure auto-layout on the set: layoutMode='VERTICAL', primaryAxisSizingMode='AUTO', counterAxisSizingMode='AUTO', itemSpacing=16, padding 16 all sides.
  4. Place at page coordinates x=-1400, y=-800 (subcomponentSetA region).
  5. Add SLOT property via cardMediaSet.addComponentProperty('Media', 'SLOT', '').
  6. Bind each variant's MediaFrame: mf.componentPropertyReferences = { slotContentId: slotProp }.

4 Card Footer Subcomponent S4

Builds the Card Footer component set — a 3-variant component (Group, Single, None) that controls the action buttons at the bottom of the card. The Group variant shows two buttons plus an overflow button. Single shows one full-width button. None shows nothing. Nested button instances are exposed so their labels and icons are editable from the parent card's property panel.

Operations

  1. Create 3 COMPONENT nodes for Pattern=Group, Pattern=Single, Pattern=None.
  2. Group variant: create ActionGroup HORIZONTAL frame with PrimaryButton, SecondaryButton (library Button instances) + OverflowButton.
  3. Single variant: one SingleButton library instance with layoutGrow=1.
  4. Call combineAsVariants, configure auto-layout, place at x=200, y=-800.
  5. Add 4 INSTANCE_SWAP properties with preferredValues pointing to the Button library set.
  6. Bind each button instance: inst.componentPropertyReferences = { mainComponent: propKey }.
  7. R25 Run exposeNestedInstances() on every variant to set isExposedInstance = true on all buttons with exposeAtParent: true in the spec.

Key Function: exposeNestedInstances()

exposeNestedInstances(componentVariant, specStructure)
Recursive — walks the spec tree depth-first
For each entry in the spec structure: if exposeAtParent === true, find the matching INSTANCE by name and set isExposedInstance = true. If the spec entry has children, recurse into them. This is general-purpose — works for any subcomponent, not just Card Footer.
function exposeNestedInstances(componentVariant, specStructure) {
  for (const [nodeName, nodeSpec] of Object.entries(specStructure)) {
    if (nodeSpec.exposeAtParent === true) {
      const inst = componentVariant.findOne(
        n => n.type === 'INSTANCE' && n.name === nodeName
      );
      if (inst) inst.isExposedInstance = true;
    }
    if (nodeSpec.children) exposeNestedInstances(componentVariant, nodeSpec.children);
  }
}

Result

Group variant: isExposedInstance = true on PrimaryButton, SecondaryButton, OverflowButton (3 instances). Single variant: SingleButton (1 instance). None: 0 instances. All read from the spec tree — no hardcoded names.

5 Parent Variant Construction S5

This is the main build phase. It creates all 24 variants of the parent component by computing the cartesian product of three axes: Thumbnail Type (4 values), Emphasis (3 values), and Density (2 values). Each variant is a fully structured card with containers, text nodes, slot frames, and anchor nodes — ready for subcomponent instances and property bindings in later phases.

Key Function: mkAutoLayoutFrame()

R26 + R28: This is the ONLY permitted method for creating auto-layout containers. No manual figma.createFrame() + layoutMode + resize() sequences.
function mkAutoLayoutFrame(name, layoutMode, parent) {
  const f = figma.createFrame();
  f.name = name;
  f.layoutMode = layoutMode;
  // R28: set sizing via properties, NEVER via resize()
  f.layoutSizingVertical = 'HUG';
  if (EDGE_TO_EDGE.has(name)) {
    f.layoutSizingHorizontal = 'FILL';
  }
  // R26: cross-axis stretch based on parent direction
  if (parent) {
    parent.appendChild(f);
    const parentMode = parent.layoutMode;
    if (EDGE_TO_EDGE.has(name)) {
      if (parentMode === 'VERTICAL')       f.layoutAlign = 'STRETCH';
      else if (parentMode === 'HORIZONTAL') f.layoutGrow = 1;
    }
  }
  return f;
}

Key Function: sizeInstance()

sizeInstance(instance, sizingSpec)
sizingSpec: { w, h, sizH, sizV } from spec.instanceDimensions
Sizes INSTANCE nodes to spec-declared fixed dimensions. resize() IS permitted here because these are FIXED-dimension nodes, not auto-layout containers. Reads from spec.instanceDimensions — never hardcoded pixel values.
function sizeInstance(instance, sizingSpec) {
  if (sizingSpec.w && sizingSpec.h) instance.resize(sizingSpec.w, sizingSpec.h);
  instance.layoutSizingHorizontal = sizingSpec.sizH;
  instance.layoutSizingVertical   = sizingSpec.sizV;
}

6 Insert Subcomponent Instances S6

Replaces the temporary anchor nodes (CardMediaAnchor and CardFooterAnchor) inside each of the 24 parent variants with real instances of the Card Media and Card Footer subcomponents built in Phases 3–4. Each instance is flagged as "exposed" so its nested properties appear in the parent card's property panel — this is how designers can change the media pattern or button labels directly from the card without diving into nested layers.

Operations

  1. Find default variants: Card Media Pattern=None, Card Footer Pattern=Group.
  2. For each of the 24 parent variants, find CardMediaAnchor and CardFooterAnchor by name.
  3. Create instances from the default variants, insert at the anchor's position in the parent, remove the anchor.
  4. Set layoutAlign = 'STRETCH' on Card Media instances.
  5. Set layoutAlign = 'STRETCH' and layoutGrow = 1 on Card Footer instances.
  6. R25 Set isExposedInstance = true on both Card Media and Card Footer instances (level 1 — parent-level subcomponent exposure).

Instance Sizing

After insertion, Card Media instances are sized per spec.instanceDimensions['Card Media'] using sizeInstance(). Dimensions vary by Thumbnail Type: Dominant = FILL×144, Primary = 128×128, Secondary = 72×72, None = FILL×HUG.

isExposedInstance: This is the only working mechanism to surface nested properties. addComponentProperty('VARIANT') and nested-key componentPropertyReferences both fail with Plugin API errors.

8 Node-to-Property Binding S8

Connects every node in every variant to its corresponding property — slots to slot properties, visibility frames to boolean toggles, and swap instances to swap properties. This is what makes the card's property panel actually control the card's content. After this phase, changing "Title" in the property panel updates the Title text node across all 24 variants.

R27: Slot Above Divider and Slot Below Divider each get BOTH their slot content binding and their visibility boolean in a single componentPropertyReferences write — because setting it twice overwrites the first binding.

Binding Categories

SLOT bindings (5 per variant): Title Thumbnail, Detail Line 1 Icon, Detail Line 2 Icon, Slot Above Divider (+ visible), Slot Below Divider (+ visible). Each via node.componentPropertyReferences = { slotContentId: propKey }.

INSTANCE_SWAP bindings (2 per variant): Card Media → Media Swap, Card Footer → Footer Swap. Each via node.componentPropertyReferences = { mainComponent: propKey }.

BOOLEAN bindings (5 per variant): Detail Lines, Detail Line 1 Row (via Detail Line 1 Visible), Detail Line 2 Row (via Detail Line 2 Visible), Divider, Action Container (via Footer). Each via node.componentPropertyReferences = { visible: propKey }.

Combined write: Slot Above/Below Divider frames must set { slotContentId: ..., visible: ... } in a SINGLE componentPropertyReferences assignment. Two separate writes would overwrite the first key.

11 Nested Button Label Exposure S11

Documents a known Plugin API limitation: button labels inside imported library Button instances cannot be exposed as flat parent-set TEXT properties. The API rejects binding componentPropertyReferences on instance sublayers. Instead, button labels are accessible through the isExposedInstance mechanism set in Phase 4 (on buttons inside Card Footer) and Phase 6 (on Card Footer inside the parent card). Designers edit button labels by expanding the Footer Swap section in the property panel.

API Limitation (R12)

Error: Cannot set component property references on instance sublayer — returned when attempting to write componentPropertyReferences on any TEXT node that is an inherited sublayer of a library instance (e.g., a button's Label text node).

Working Substitute

The isExposedInstance chain: Phase 4 exposes button instances inside Card Footer (R25 recursive) → Phase 6 exposes Card Footer instances inside the parent card (R25 level 1). This two-level chain surfaces every nested button property (Label, Left Icon, Style, etc.) under Footer Swap in the parent's property panel, with zero orphan parent props.

Tradeoff

The only alternative — rebuilding Button from scratch with native TEXT children inside Card Footer — would sacrifice theme-mode inheritance and library updates. The pipeline preserves library Button.

R25 Recursive Audit Function

function auditExposure(component, specStructure, errors) {
  for (const [nodeName, nodeSpec] of Object.entries(specStructure)) {
    if (nodeSpec.exposeAtParent === true) {
      const inst = component.findOne(
        n => n.type === 'INSTANCE' && n.name === nodeName
      );
      if (!inst) errors.push(`Missing INSTANCE '${nodeName}'`);
      else if (!inst.isExposedInstance)
        errors.push(`${nodeName}: isExposedInstance !== true`);
    }
    if (nodeSpec.children) auditExposure(component, nodeSpec.children, errors);
  }
}

Semantic Surface System

The system separates what a surface is (semantic role) from where it appears (environment level), so the same token tells a designer the purpose while automatically adjusting tonal value based on context.

Monocore Semantic 24 surface roles 4 modes Collection: Monocore Semantic Updated May 2026
Concept

Architecture #

A semantic role describes the purpose of a surface and stays stable across all environments and themes. An environment level describes whether a surface sits in the root app or inside an elevated container. The two axes are orthogonal: pick the role for intent, and the parent frame's mode resolves the role to the right tonal value automatically.

Axis 1 — The "what"

Semantic Surface Roles

Stable, purpose-named tokens like Surface/Base or Surface/Floating. The role never changes regardless of theme or environment.

Axis 2 — The "where"

Environment Levels

L1 is the root application environment. L2 is an elevated canvas — bottom sheets, modals, drawers. The parent frame's mode controls which value the role resolves to.

Shape & Spacing #

Two parallel scales govern how surfaces meet the eye: the radius applied to their corners, and the space between them. Both follow the same naming philosophy as the surface system — group by meaning, name by role, so the value behind a token can change in one place.

Rules

Behavioral Rules #

Five rules govern how the system behaves. Each one closes off a class of mistake the design system has historically made: arbitrary elevation steps, ad-hoc semantics per component, and value drift when context changes.

Semantic stability. A semantic role's meaning never changes. Surface/Floating always means "floating temporary UI surface" — only its tonal value shifts between L1 and L2.
Primary means "always extreme," not "always lighter." Unlike Surface/Canvas, which adapts to its container, Surface/Primary always holds the extreme value (pure white in light, pure black in dark) regardless of elevation. On a Base or Raised container it reads as lifted (lighter in dark mode). But inside a dark Overlay (Grey/10), a Primary card goes to Grey/0 — darker than its container, an intentional inversion. Use it for cards and contained regions that must maintain maximum contrast against their surround rather than blending into the elevation hierarchy.
No new elevations. The system has L1 and L2, and only L1 and L2. Floating is its own thing, not L3.
Floating is not elevation. Menus, popovers, tooltips, and reaction pickers use Surface/Floating regardless of whether they appear in an L1 or L2 parent context. They are transient, not canvas-elevated.
Platform-chrome surfaces are exceptions. Surface/Dominant (top) and Surface/Quaternary (bottom tabs) exist outside the canonical L1/L2 system to give the OS room to impose its own materials. Their declared values are fallbacks — actual rendering may be overridden by iOS 26 large-title materials or Android tab-bar blur on supported devices.
Overlay forces L2. Presenting a modal, sheet, or drawer shifts the parent frame from L1 to L2. This means the apparent Dark L1 value collision between Surface/Overlay and Surface/RaisedSecondary (both resolve to Grey/10) is never actually rendered — by the time Surface/Overlay is on screen, the surrounding tokens have already resolved to their L2 values, where RaisedSecondary becomes Grey/14 and the visual separation returns.
A handful of tokens differ between L1 and L2. The rest resolve to the same value across environment levels. The L1/L2 distinction matters most for: Floating, Sunken, Interactive. (Base and RaisedSecondary share a single value across both L1 and L2 — the environment shift no longer applies to them.)
Workflow

How Designers Use It #

The system reduces every surface decision to two choices: pick the role, set the parent's mode. Everything else resolves automatically.

  1. Pick the semantic role that describes intent (Surface/Base, Surface/Floating, etc.).
  2. Bind the variable to a fill or stroke.
  3. Set the parent frame's mode (Light L1, Light L2, Dark L1, Dark L2).
  4. The variable resolves to the correct tonal value automatically.

When entering an elevated context (sheet, modal, drawer), the parent frame's mode switches from L1 to L2 — every surface inside picks up the L2 tonal values without changing token names.

Key Decisions #

Two mappings were corrected from name-similarity to purpose. The stroke layer was rebuilt as a three-token ladder so adjacent layers don't blend and designers don't have to keep nudging values for every new UI combination.

Corrections

  • Subtle Fill / Subtle Base → Base (not Subtle). The old "Subtle" labels described a contained surface, not an alpha overlay — the purpose belongs to Surface/Base.
  • Border Dominant → Stroke/Default (not Stroke/Elevated). It's the standard separation stroke, not the edge of a floating surface.
  • Tertiary → Overlay: a clean mapping. Sheets, popovers, and modals are all modal/blocking surfaces and share the same role.

Theming

Theming lets a product swap its brand color set without touching any component. The five Brand/Accent * tokens are the semantic layer products bind to; the values resolve from the active brand palette in Global colors/Brand light and Brand dark, which themselves are owned by the Brand collection and can be repointed per product.

5 accent roles 9 modes Brand-swappable
Concept

How theming works #

The Accent ladder describes the role of a branded surface or fill — "most prominent," "standard," "subtle," and so on — without committing to a specific color. The active brand palette (set by the product) supplies the underlying hex values. Switch the palette and every Accent consumer follows; rename the role and you've made a single coordinated update across every consuming component.

Semantic ladder, palette-driven values. Components bind to Brand/Accent Primary, not to Brand light/Primary. The active palette decides what "Primary" resolves to. A Teams build can swap the palette without recompiling components.

Across modes #

Accent tokens are mode-aware in the same sense as the Surface ladder. Workflow

Swapping the active palette #

A product can ship multiple brand palettes (e.g. Teams default, partner co-brand, seasonal palette) and swap the active one at runtime. The Accent tokens never change; the palette they alias to does.

  2. Bind components to Brand/Accent *. Components never reference the palette directly.
  3. At runtime, point the Accent tokens at the chosen palette. Every consumer updates without recompiling.
  4. Add new palettes by extending the Brand collection. The Accent ladder doesn't change shape.

Scaling Rules

A two-part framework for Dynamic Type. First, classify each component into a scale tier that sets how much its text is allowed to grow (Scale0, Scale1, ScaleMax). Then, at the upper bound of that tier, use reflow tokens to decide what happens to elements that still won't fit — extending vertically (extend0–extendFit) or stacking columns (virticalStack).

3 scale tiers 5 reflow tokens iOS & Android
Concept

Why a scaling framework #

Dynamic Type lets users push text to extreme sizes for accessibility. Without intervention, every text element scales freely — and components collapse: timestamps collide with labels, tab labels wrap to three lines, search placeholders shove their icons off-screen. The framework gives designers two levers — a per-component scale tier and per-element reflow tokens — so a chat row can stay legible at max zoom without breaking the rest of the screen.

Tune at the cap, not at every step. Designers configure behavior at one point — the upper bound of the component's scale tier. The framework interpolates every intermediate Dynamic Type size between the default layout and the configured max-zoom layout.
Foundations

Scale tiers #

Each component belongs to one of three tiers based on how much its text is allowed to grow. The tier sets the ceiling; the reflow tokens decide what happens at that ceiling.

TierMaximum scaleUse it for
Scale01× · no scaleComponents whose text content is always the same and lives in the same location. Tab bars, fixed labels in chrome that the user reads at a glance, content anchored to a UI region rather than to body text.
Scale11.3× · capped scalingComponents whose text lives in the same location and doesn't change often. Headers, search placeholders, badges, token fields, message-compose chrome — text that's allowed to grow somewhat while the surrounding container stays usable.
ScaleMaxUnrestrictedComponents whose text is the user's content — chat list items, message bodies, anything the user reads attentively. Scale continues to the full Dynamic Type ceiling; reflow tokens take over from there.
Pick the tier from the role, not the size. The question is "what does this text do?" — chrome (Scale0), navigation or container labels (Scale1), user content (ScaleMax) — not "how big do I want it to be."
Reflow

Reflow tokens — extending lines #

At the upper bound of Scale1 and ScaleMax — the max zoom level where comprehension is already constrained — four extend tokens control how many additional lines a text element is allowed to wrap to before truncating. Assign the token per text element; the cell as a whole inherits the most permissive value of its children.

TokenValueBehavior
extend0Plus 0No lines added. The text truncates at its original line count.
extend1Plus 1Add one more line before truncating.
extend2Plus 2Add two more lines before truncating.
extendFitUnboundNever truncate. Let all text reflow into as many lines as it needs.
Pick the smallest extend that preserves the message. A timestamp doesn't need extendFit; a primary message label might. Defaulting every element to extendFit defeats the framework — every cell takes unbounded vertical space and pushes the next cell off-screen.
Reflow

Reflow tokens — stacking columns #

When the extend tokens still aren't enough — typically because the leading column has grown so wide it begins to clip the trailing column — the trailing column reflows beneath the leading column instead of competing with it for horizontal space.

TokenBehavior
virticalStack (sic — token name as published)When the leading column begins to clip the trailing column, reflow the trailing column under the leading column.

This gives each cell two degrees of freedom at max zoom: vertical expansion within the original row (the extend tokens), and column-stacking once horizontal space is exhausted (virticalStack). Together they let a chat row with a long message and a long timestamp degrade gracefully instead of overlapping.

The order of fallback. The extend tokens are tried first — they preserve the two-column layout. virticalStack is the second step, only reached when the trailing column still can't fit. Two cascading fallbacks, applied in sequence by the framework.
Workflow

Applying the framework #

Two decisions per component — the tier and the per-element reflow tokens. Everything in between is interpolated.

  1. Pick the tier for the component as a whole. Is it chrome (Scale0), navigation/container (Scale1), or user content (ScaleMax)?
  2. For Scale1 and ScaleMax, assign an extend token to each text element based on how much of it must remain readable at the cap.
  3. Assign virticalStack to the trailing column if it still clips after the extends are applied.
  4. The framework handles the rest. Intermediate sizes interpolate between the default layout and the cap layout you've configured. No per-step authoring required.

Platform implementation

iOS implementation #

iOS doesn't use a decimal scaling factor. Each component opts into one of four UIContentSizeCategory strategies that govern the band of allowed text sizes under the user's Dynamic Type setting. The framework tiers (Scale0, Scale1, ScaleMax) map onto these strategies as follows.

The four iOS strategies

Each strategy defines a min and max content size category. Sizes outside the band are clamped to the bound. The five accessibility categories (AX1–AX5) are only reachable when the user enables Larger Accessibility Sizes in iOS Settings > Accessibility > Display & Text Size.

iOS strategy Range What it does
default xSmall → AX5 Follows the user's device setting with no upper bound. Text scales through every category including all five accessibility sizes.
large xSmall → Large Min is unbounded; max preferred size is Large (the system default). Components grow from the smallest scale up to Large and stop.
excludeAccessibilityCategory xSmall → xxxLarge Caps at xxxLarge — the largest non-accessibility size. AX1–AX5 never reshape the layout.
largeToExcludeAccessibilityCategory Large → xxxLarge Floored at Large, capped at xxxLarge. Text never goes below the system default and never enters the accessibility range.

  3. Collect all request blocks into a ## ⚠ Open Requests to Designer section at the TOP of the spec, so reviewers see them before reading the body.
  4. Add a one-line status at the top: Spec completeness: N/M categories resolved.
  5. If N < M, append: "This spec is INCOMPLETE and cannot drive the Monocore build pipeline until all open requests are resolved."
W_SPEC_MISSING_CATEGORY — blocking. A required category has no concrete value. The pipeline's Phase 0c will halt.
W_SPEC_FABRICATED_VALUE — blocking. The generator included a value that does not trace to source code, a design token file, or an explicit designer annotation. Remove the value, emit a REQUEST_FROM_DESIGNER block instead. Never ship a spec with un-sourced values.

Nested Element Catalog

The exhaustive list of base components and controls that component-spec-generation.rules.json Step 8 requires the generator to document whenever they appear as nested elements inside a parent component. 72 elements across 8 categories.

Rule file step 8 72 elements 8 categories Paired with per_element_documentation

How this catalog is used

Step 8 of the spec generation process requires the generator to produce a full sub-section for every non-text child element inside the parent component — buttons, dividers, chips, avatars, toggles, anything that isn’t plain text. The per-element sub-section is defined in per_element_documentation (identity, variants/properties, sizing/layout, container composition, visibility, icon/accessory support).

This catalog is the companion lookup. Once the generator identifies an element’s kind, the document column below lists the specific attributes that matter for that kind. A Button needs emphasis variant, size, enabled state, leading/trailing icon, sizing mode, visibility flag. A Date Picker needs single/range mode, min/max date, format, locale, inline vs popover. This is what keeps coverage uniform across every nested element, not just the obvious ones.

If the element type isn’t in this list, fall back to the Custom / Unknown entry — apply the same six per-element sub-sections anyway, pull the library component key if the element is from the library, or flag it for potential library migration under R4.

Design System Provenance

Canonical Source Libraries

Every visual element kind has exactly one authoritative Figma library. The pipeline imports from that library and is forbidden from fabricating equivalents. Enforced by R41 and audited at every phase.

rules/libraries.json v2.0 R41_CANONICAL_SOURCE_LIBRARY_ENFORCEMENT 4 libraries 4 element kinds mapped

Why canonical sources matter

For any design system, each kind of element must have a single declared source of truth. Otherwise the pipeline has room to fabricate: draw an overflow-icon-shaped vector because the glyph wasn’t imported, write rgb(1,1,1) as a raw fill because no variable was bound, re-implement a library Divider as a 1-px frame. These fabrications look correct on first render and drift the moment the design system updates.

R41 closes this surface. rules/libraries.json declares the authoritative library for every element kind. The pipeline halts at Phase 0c if an element kind is used without an assignment, at Phase 1 if a library import fails, at Phase 5 if a raw RGBA fill or ad-hoc vector is written where a library asset should be, and at Phase 12 if any node in the built set traces back to the wrong library.

R41: Canonical Source Library Enforcement

The obligation lives in monocore-component-design.rules.json as R41_CANONICAL_SOURCE_LIBRARY_ENFORCEMENT. It fires at five enforcement points.

PhaseCheckError code on failure
Phase 0c (spec completeness)Spec declares element_source_map coverage for every element kind it usesE_SOURCE_LIBRARY_NOT_DECLARED
Phase 1 (library imports)importComponentByKeyAsync / importVariableByKeyAsync succeeds for every referenced roleE_CANONICAL_LIBRARY_IMPORT_FAILED
Phase 5 (variant build)No createVector() for iconic nodes; no raw RGBA where a token variable existsE_NONCANONICAL_VISUAL_VALUE

On missing asset: if a required glyph, color, or component is absent from its canonical library, the pipeline HALTS. It does not substitute a near match from another library, does not fabricate a local equivalent, and does not silently proceed. The designer adds the asset upstream, then the build re-runs.
